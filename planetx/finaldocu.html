Concept and Inspiration:
Planet X is created based on the concept of Will Android Dream of Electronic Sheep? It aims to enable players to explore the notion of balance in the world we live in, in spite of its background set in A.C. 2049 in a parallel university from our world.

On Planet X, there are three kinds of resources: TREE, WATER and ATOMIC ENERGY to be utilized. After the energy they respectively accumulated reach a balance, there will be Planet X inhabitants coming out. Once the balance is off, for example there are too much waste accumulated that it becomes detrimental to the inhabitants, there will be no more babies born. The moment that the population declines to 0, Planet X becomes dead again, and player will be noticed of the time duration that they brought Planet X back to life, as well as the total population that were born into the Planet X.

Challenges:
- Typing effect and corresponding sound effect.
Solved it by: putting all the words that need to be displayed in a very long string, with all of the blank space and line feed in it. Split the string into array, and display the array content. Sound plays a very important rule in setting the whole feeling and atmosphere, so I edited a found typewriter soundtrack and made it pause corresponding to the typewriter visual effect.

- Visual of forest/water/nuclear area evolve.
Setting the initial value that the cell of water source, forest and nuclear has, the flow amount that each cell passes on to its 4 neighbor cells. During the process of resource transporting, there is some amount of them getting evaporated. The use of HSB color mode enables the lightness of every grid to grow proportionally with the dominant amount of the element out of four (natural energy, nuclear energy, sanitation, nuclear waste).

- The legend (indicator)
Created using vertex(), beginShape() and endShape(), and map the values onto the axis.

Lessons to take away:
I. Always start with the basic logic first and then get down to building the game technically.



II. A good game should be complete and intuitive.
Complete means that it has a solid playable version with an opening and ending; Intuitive means that the interaction required of the player needs to be intuitive.

III. Naming of variables should be as organized as possible.
I can anticipate that when building a very large and complicated game, variable names could be easily mistaken, which could be the potential reason for bugs in the program.

IV. 90-90 rules.
Although this is a universal rule in computer science field, I still feel the need to emphasis it here. There could be sometime when it seems that the project is almost done, but in the end the rest of the things will still take up a big chunk of your time to finish. Therefore, it is always a good idea to start early.

V. Run enough user tests in order to improve the game.
As designers/developers, we tend to ignore places that are not intuitive or have other problems. So we need another pair of eyes to inspect the game to make it into a better one.

The problem now is that there seems to be some problem with color. For the convenience of debugging, I set the parameters of water, forest and nuclear all to the same value, yet on the screen the nuclear waste seems to be conquering all over the canvas. I was not sure whether it was because I mapped the value to the opacity of each area's designated color and for different color they have different visibility on black even when their opacity are the same, for example (assuming) (255,0,0,0.5) is easier to be distinguished against black than (0,255,0,0.5);

I solved the problem by switching color mode to HSB. Now I only need to map the values proportional to the "B" value of each area. Now the landscape looks like this.
